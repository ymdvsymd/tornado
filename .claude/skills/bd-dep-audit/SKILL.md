# bd-dep-audit: bd チケット依存関係監査

全 active チケットの依存関係を `rules/bd-dependency-protocol.md` に基づいて監査し、
コードベース検証 → ファイル重複分析 → 依存設定 → チケット更新を一貫実行する。

チケット作成後の事後チェックとして使う。作成時の依存分析は各 skill（moonbit-audit, log-audit, plan-to-beads 等）が
bd-dependency-protocol に従って実施するが、このスキルはそれらが見落とした依存や、
コードベース変更によって陳腐化したチケット内容を検出する。

## 引数

```
/bd-dep-audit [--dry-run] [--ticket=<id1>,<id2>,...] [--skip-verify]
```

- `--dry-run`: レポート出力のみ。`bd dep add` / `bd close` / `bd update` を実行しない
- `--ticket=<ids>`: 指定チケットのみ分析（カンマ区切り）。省略時は全 active チケット
- `--skip-verify`: Phase 2（コードベース検証）をスキップ。高速だが無効チケットを見逃す

`$ARGUMENTS` の内容: `$ARGUMENTS`

---

## Phase 1: チケット収集

1. `$ARGUMENTS` を解析し、オプションを抽出する
2. 対象チケットを取得する:

```bash
bd list --status=open
bd list --status=in_progress
```

3. 各チケットの詳細を `bd show <id>` で取得する
   - 並列で取得（`bd show` を複数同時実行）
4. 各チケットから以下を抽出する:

| 抽出項目 | 抽出元 |
|---------|-------|
| 対象ファイル一覧 | description 内のファイルパス（バッククォート内 `src/...` 等） |
| 変更種別 | type（bug/feature/task）と description の内容から判定 |
| 行番号参照 | description 内の `L数字` パターン |
| 既存の依存関係 | `bd show` 出力の dependencies/blocked-by セクション |

5. 既存の依存関係を記録する。Phase 4 で重複設定を避けるために使用する。

**出力**: チケット情報テーブルをユーザーに表示する:

```markdown
## チケット一覧（N 件）

| ID | P | Type | タイトル | 対象ファイル | 既存依存 |
|----|---|------|---------|------------|---------|
| whirlwind-xxx | P2 | bug | ... | src/ralph/planner.mbt | なし |
| whirlwind-yyy | P3 | task | ... | src/ralph/ralph_loop.mbt | blocked-by: xxx |
```

---

## Phase 2: コードベース検証（CRITICAL）

チケットの description が実際のコードベースと一致しているかを検証する。
このフェーズが依存分析の精度を決定する。省略すると無効チケットに基づいた誤った依存関係を設定してしまう。

`--skip-verify` 指定時はこのフェーズをスキップする。

### 検証手順

各チケットについて、Explore エージェントを起動して並列検証する（最大 3 並列）:

1. **ファイル存在チェック**: description に記載された対象ファイルが実在するか（Glob で確認）
2. **行番号・関数参照チェック**: 記載された行番号に該当する関数・コードが存在するか（Read で確認）
3. **主張内容の真偽チェック**: チケットの前提が今も成立するかをコードで検証する

主張内容の真偽チェックが最も重要。具体例:
- 「doc コメントが欠落」→ 対象関数の直前に `///|` が本当にないか Read で確認
- 「ワイルドカード `_` を使用」→ 対象の match 式に `_` が本当にあるか確認
- 「テストファイルが存在しない」→ Glob で本当にないか確認
- 「let mut が不要」→ 対象行で本当に `let mut` が使われているか確認

### 検証の判定基準

**「チケットの主張が事実と一致するか」** で判定する。チケットが望む最終状態ではなく、
チケットの前提（＝変更が必要な理由）が事実かどうかが基準。

- チケットが「ファイル X が存在しない」と主張 → X が不在なら **valid**（チケットは作成を要求している）
- チケットが「関数 Y に doc コメントがない」と主張 → Y に doc コメントがあれば **invalid**（既に解決済み）

Explore エージェントへのプロンプトにはこの判定基準を明示し、
「チケットが解決しようとしている問題が今も存在するか？」と問うこと。

### 検証結果の分類

| 分類 | 基準 | アクション |
|-----|------|----------|
| **valid** | 全主張がコードベースと一致 | そのまま依存分析に進む |
| **stale** | ファイル/行番号がずれているが主張は概ね正しい | 修正候補としてレポート |
| **invalid** | 主張が事実と異なる（既に修正済み、参照先が誤り等） | クローズ候補としてレポート |

### 出力

検証結果テーブルをユーザーに表示する:

```markdown
## コードベース検証結果

| チケット | ステータス | 詳細 |
|---------|----------|------|
| whirlwind-xxx | valid | 全主張を確認 |
| whirlwind-yyy | invalid | 対象関数に既に doc コメントあり。行番号参照も誤り |
| whirlwind-zzz | stale | 行番号が 10 行ずれ（L100→L110）。関数は存在 |
```

**invalid チケットが見つかった場合**: ユーザーに確認し、承認後にクローズする。
invalid チケットは Phase 3 のファイル重複分析から除外する。

---

## Phase 3: ファイル重複マトリクス構築

Phase 1 で抽出した対象ファイル一覧を使い、全チケットペアのファイル重複を検出する。

### 3-1. ペア生成

N 件のチケットから N*(N-1)/2 のペアを生成する。
チケット数が多い場合（20+）、まずファイルパスの完全一致でフィルタし、一致なしのペアをスキップする。

### 3-2. 重複判定

各ペアについて:

| 判定 | 基準 |
|-----|------|
| **ファイル重複** | 対象ファイルが 1 つ以上一致 |
| **関数重複** | 同一ファイル内で同一関数（名前 or 行番号範囲の重複） |
| **モジュール重複** | 同一パッケージ（`src/ralph/` 等）だがファイルは異なる |

ファイル重複のみを依存関係の根拠とする。モジュール重複は参考情報として記載するが、
依存関係は設定しない（同パッケージでもファイルが異なれば通常は独立）。

### 3-3. 既存依存のフィルタ

Phase 1 で取得した既存の依存関係と照合し、既に設定済みのペアは「既存依存（変更なし）」として記録する。
新規に検出されたファイル重複のみ Phase 4 に進む。

### 出力

```markdown
## ファイル重複分析（N*(N-1)/2 ペア中 M ペアで検出）

| ペア | 重複ファイル | 状態 |
|-----|------------|------|
| whirlwind-870 ↔ whirlwind-4th | src/ralph/ralph_loop.mbt | **既存依存** (870 blocks 4th) |
| whirlwind-xxx ↔ whirlwind-yyy | src/ralph/verifier.mbt | **新規検出** → Phase 4 へ |

検討したが独立と判定:
- whirlwind-aaa ↔ whirlwind-bbb: 同一ファイル src/x.mbt だが編集箇所が 150 行離れており別関数
（該当なしの場合も「検討したが独立と判定: なし」と明記する）

モジュール重複（参考情報、依存設定なし）:
- whirlwind-eae, whirlwind-870: 同一パッケージ src/ralph/ だがファイルは異なる（planner.mbt vs ralph_loop.mbt）
（該当なしの場合も「モジュール重複: なし」と明記する）
```

## Phase 4: 依存方向の決定

ファイル重複が検出されたペアについて、`rules/bd-dependency-protocol.md` の
Step 3（依存方向ヒューリスティクス）を適用する。

### ヒューリスティクスの適用順序

1. **ファイル依存**: import/依存関係がある場合、依存元が先
2. **変更種別階層**: `インフラ/設定 > 型定義/インターフェース > 実装 > テスト > ドキュメント`
3. **明示的順序**: ステップ番号、"after"/"requires" 等のキーワード
4. **実務的コスト**: 両方向を比較し、依存設定後のチケット修正コストが低い方を選択

「実務的コスト」は依存設定の方向によって、どちらのチケットの description を書き換える
必要があるかを考慮する。たとえば、大規模なファイル分割チケット A と、
A の対象ファイル内の関数を修正するチケット B がある場合:

- B → A の順（B が先）: B は現在のファイルパスで正しく動作。A の行番号が若干ずれるが関数グループは有効
- A → B の順（A が先）: B の description のファイルパス・行番号を全面書き換え。修正コスト大

この場合、修正コストの低い「B が先」を選択し、A の description に行番号変動の注記を追加する。

### ファイル重複があっても依存不要なケース

同一ファイルでも、編集箇所が十分に離れている場合（100 行以上）かつ
異なる関数を対象とする場合は、依存関係を設定しない。
この判断を「検討したが独立と判定」としてレポートに記載する。

## Phase 5: 確認テーブルとユーザー承認

検出された全ての変更をユーザーに提示する。

### 5-1. 無効チケットのクローズ（Phase 2 の結果）

```markdown
## クローズ対象チケット

| チケット | 理由 |
|---------|------|
| <id>: <title> | 対象関数に既に doc コメントあり。行番号参照も誤り |

クローズしますか？ (yes / 編集 / no)
```

### 5-1b. Stale チケットの修正提案（Phase 2 の結果）

```markdown
## Stale チケット（行番号ずれ等）

| チケット | 問題 | 修正案 |
|---------|------|-------|
| <id>: <title> | L100 → L110（10行ずれ）。関数 foo は存在 | description の行番号を更新 |

修正しますか？ (yes / 編集 / no)
```

### 5-2. 依存関係の設定（Phase 4 の結果）

```markdown
## 依存関係の設定

| Blocked チケット | Blocked by | 根拠 |
|-----------------|-----------|------|
| <id>: <title-A> | <id>: <title-B> | 同一ファイル: src/x.mbt。B の実装変更を先に行い、A の分割で反映 |

依存関係を設定しますか？ (yes / 編集 / no)
```

### 5-3. チケット内容の更新（依存設定に伴う修正）

依存関係を設定するチケットの description と notes を更新する必要がある:

```markdown
## チケット更新

| チケット | 更新内容 |
|---------|---------|
| <blocked-id> | description: 行番号変動の注記追加。notes: 依存先チケット ID と理由を追記 |
| <blocker-id> | notes: ブロック対象チケット ID を追記 |

更新しますか？ (yes / 編集 / no)
```

### 5-4. 独立チケットのサマリー

```markdown
## 独立チケット（依存関係なし）

- <id>: <title> — <reason>（例: 他チケットとファイル重複なし）
```

---

## Phase 6: 実行

ユーザー承認後に変更を実行する。`--dry-run` の場合はスキップ。

### 実行順序

1. 無効チケットのクローズ（`bd close <id> --reason="..."`)
2. 依存関係の設定（`bd dep add <blocked> <blocker>`）
3. チケット内容の更新（`bd update <id> --description/--notes`）

### 実行後の検証

```bash
bd blocked              # blocked チケットの確認
bd ready                # ready チケットが正しいか確認
bd show <updated-id>    # 更新内容の確認
```

### 完了サマリー

```markdown
## 監査完了

| アクション | 件数 |
|-----------|-----|
| クローズ | N 件 |
| 依存関係設定 | N 件 |
| チケット更新 | N 件 |
| 独立（変更なし） | N 件 |

現在の状態: X 件 open（Y 件 blocked）、Z 件 ready
```

---

## 注意事項

- Phase 2（コードベース検証）は時間がかかるが、依存分析の精度に直結する。
  `--skip-verify` は「チケットが最近作成されたばかりで内容が最新」と確信がある場合のみ使用する
- 依存関係の設定は `rules/bd-dependency-protocol.md` Step 4 のユーザー承認フローに従う。
  自動実行しない
- チケットの priority を変更する権限はこのスキルにはない。
  P4 チケットが P3 チケットをブロックする優先度逆転を検出した場合はレポートに記載する
- 循環依存を検出した場合はエラーとして報告し、ユーザーに解決を求める
- 全チケットが valid かつ独立（ファイル重複なし、既存依存で網羅済み）の場合、
  「監査完了: 変更不要。全チケット valid、依存関係は最新」と報告して終了する
